# Phase1 交付接口规范

> 学生可以自定主题，可以不跑参考答案的代码，但**每天必须交出这几件固定名称的文件**。
> 这些文件就是"接口"——无论主题怎么变，文件名和格式不变，Day2 才能读 Day1 的产出，Day3 才能读 Day2 的产出。

---

## Day1 · 模型路由

**核心产出：一张路由表，回答"每个任务该用哪个模型"**

| 交付物 | 文件名 | 格式 | 谁用 |
|---|---|---|---|
| 路由决策 | `day1_to_day2_brief.json` | JSON | → Day2 读取 |
| 原始评测数据 | `day1_results.json` | JSON | 评估用 |
| 评测结果汇总 | `model_capability_matrix.json` | JSON | 展示用 |
| 失败样本 | `failure_casebook.md` | Markdown | 展示用 |

**路由表结构（`day1_to_day2_brief.json`）：**
```json
{
  "router": {
    "product_planner": {
      "preferred_model": "模型A",
      "score": 4,
      "fallback_rule": "失败时切换到模型B"
    },
    "context_designer": { ... },
    "schema_generator": { ... },
    "code_assistant": { ... },
    "risk_checker": { ... },
    "multimodal_context": { ... }
  }
}
```

> **字段约束：**
> - `router` 的 6 个 key 名（product_planner / context_designer / schema_generator / code_assistant / risk_checker / multimodal_context）**必须一致**——Day2 硬编码了这些 key。
> - 每个 key 下 `preferred_model` 字段**必填**（链路必需）。`score` 和 `fallback_rule` 建议保留但不强制。
> - 学生可以自定义 `evidence_task`（只要 Day1 的 task_id 对得上）。`purpose`、`product_direction`、`day2_requirements` 字段可以改、可以删——不影响链路。

**原始评测数据（`day1_results.json`）：**
> 每条记录只需保留 `task.id`、`task.title`、`model.label`、`model.id`、`score`、`ok`、`output.text`。其他字段（latency、usage 等）可选。

学生可以不跑 `day1_arena_builder.py`，但**必须自己填出 `day1_results.json` 和 `day1_to_day2_brief.json`**。## Day2 · 能力定义

**核心产出：一组 JSON 文件，定义 Day3 要用到的"4 个能力"**

| 交付物 | 文件名 | 格式 | 谁用 |
|---|---|---|---|
| 产品规格 | `product_spec.json` | JSON | 展示用 |
| Prompt 库 | `prompt_library.json` | JSON | 展示用 |
| 评测用例 | `eval_cases.json` | JSON | 自检用 |
| **Agent 合同** | `agent_handoff/day2_agent_contract.json` | JSON | → Day3 读取 |
| 复习卡 Schema | `schemas/review_card.schema.json` | JSON | 格式约束 |
| 答题评分 Schema | `schemas/answer_grading.schema.json` | JSON | 格式约束 |
| 多模态上下文 Schema | `schemas/multimodal_observation.schema.json` | JSON | 格式约束 |

**Agent 合同结构（`day2_agent_contract.json`）：**
```json
{
  "engine_name": "你的产品名",
  "actions": [
    {
      "name": "generate_review_cards",
      "input": "course_context: markdown/text",
      "output": "review_card[]",
      "writes": "cards/review_cards.json"
    },
    {
      "name": "grade_answer",
      "input": "question + expected_keywords + student_answer",
      "output": "answer_grading",
      "writes": "evidence/grading_result.json"
    },
    {
      "name": "extract_multimodal_context",
      "input": "input_type + image_or_screenshot_description",
      "output": "multimodal_observation",
      "writes": "knowledge/multimodal_observation.json"
    },
    {
      "name": "prepare_device_reply",
      "input": "intent + latest_agent_state",
      "output": "short_text",
      "writes": "latest_device_reply.txt"
    }
  ],
  "files": { ... },
  "day3_contract": {
    "read_before_run": [ ... ],
    "use_in_agent": [ ... ]
  }
}
```

学生可以不改 `pocket_review_engine.py`，但**必须自己填出这个 JSON**。
> **字段约束：**
> - `actions` 数组至少包含 4 个 action，name 必须为 `generate_review_cards` / `grade_answer` / `extract_multimodal_context` / `prepare_device_reply`（**必填**——Day3 硬编码了这些 action 名）。
> - 每个 action 的 `input`、`output`、`writes` 字段**必填**。`writes` 路径必须与 Day3 的 workspace 子目录对应。
> - `engine_name`、`product_name` 可以自定义。`day3_contract` 的 `read_before_run` 路径建议保留。
> - 4 个 Schema 的文件名必须一致（review_card.schema.json / answer_grading.schema.json / multimodal_observation.schema.json / agent_handoff.schema.json），schema 内部的字段学生可以自己定义。

学生可以不改 `pocket_review_engine.py`，但**必须自己填出这个 JSON**。

## Day3 · Agent 执行

**核心产出：Agent 读完 Day2 合同后，在 workspace 里生成这些文件**

| 交付物 | 文件路径 | 格式 | 谁用 |
|---|---|---|---|
| 课程摘要 | `agent_workspace/summaries/course_summary.md` | Markdown | → Day4 读取 |
| 待办 | `agent_workspace/summaries/tasks.md` | Markdown | 展示用 |
| 风险 | `agent_workspace/summaries/risks.md` | Markdown | 展示用 |
| 复习卡 | `agent_workspace/cards/review_cards.json` | JSON | → Day4 读取 |
| 知识库索引 | `agent_workspace/knowledge/chunk_index.json` | JSON | 证据 |
| 检索结果 | `agent_workspace/knowledge/retrieval_result.json` | JSON | 证据 |
| 执行记录 | `trace_log.jsonl` | JSONL | 评估用 |

学生可以自己写 Agent，也可以改 `day3_agent_core.py`，但**输出文件名和路径必须一致**。

## Day4 · 设备接入

**核心产出：Day3 Agent 包装成服务，设备触发后生成的文件和 Day3 一致**

| 交付物 | 文件路径 | 格式 |
|---|---|---|
| 设备触发后的摘要 | `day4_agent_workspace/summaries/course_summary.md` | Markdown |
| 设备触发后的待办 | `day4_agent_workspace/summaries/tasks.md` | Markdown |
| 设备触发后的风险 | `day4_agent_workspace/summaries/risks.md` | Markdown |
| 设备触发后的复习卡 | `day4_agent_workspace/cards/review_cards.json` | JSON |
| 执行记录 | `day4_trace_log.jsonl` | JSONL |

---

## 总结：接口即边界

| 天 | 核心接口（必须交的） | 递交给谁 |
|---|---|---|
| Day1 | `day1_to_day2_brief.json` | → Day2 |
| Day2 | `agent_handoff/day2_agent_contract.json` + 4 个 Schema | → Day3 |
| Day3 | `agent_workspace/cards/review_cards.json` + `summaries/` | → Day4 |
| Day4 | 同 Day3，路径改 `day4_agent_workspace/` | 最终演示 |

学生可以换主题、换模型、自己写代码、甚至用自己的硬件——**只要这 4 个接口文件的名字和格式不变，整个链条就能跑通。**
